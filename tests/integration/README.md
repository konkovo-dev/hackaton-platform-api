# Интеграционные тесты

Комплексные интеграционные тесты для Hackathon Platform API с **полным покрытием** всех REST API эндпоинтов.

## Обзор

Эти тесты проверяют работу всего API, делая реальные HTTP-запросы к запущенному инстансу. Никаких моков, никакой подготовки базы данных - тесты взаимодействуют с API точно так же, как это делал бы реальный клиент.

**Общее покрытие**: 80+ тест-кейсов для всех сервисов, основанных на официальных REST API гайдах.

## Запуск тестов

### Предварительные требования

1. **API должно быть запущено**:
   ```bash
   # Локальная разработка
   docker compose -f deployments/docker-compose.yml up -d
   
   # Или тестирование против развёрнутого API
   export API_BASE_URL=http://your-server:8080
   ```

2. **Go 1.24+** установлен

### Запуск всех тестов

```bash
cd tests/integration
go test -v
```

### Запуск конкретного тестового файла

```bash
# Только auth тесты
go test -v -run TestRegister

# Только identity тесты
go test -v -run TestGetMe

# Только hackathon тесты
go test -v -run TestCreateHackathon

# Только staff тесты
go test -v -run TestListHackathonStaff

# Только E2E тест
go test -v -run TestFullHackathonFlow
```

### Запуск конкретного теста

```bash
# Один тест
go test -v -run TestRegisterWithIdempotencyKey

# По паттерну
go test -v -run "TestList.*"
```

### Параллельное выполнение

Тесты спроектированы независимыми и могут выполняться параллельно:

```bash
go test -v -parallel 4
```

### Тестирование против удалённого API

```bash
# Установите URL базового API
export API_BASE_URL=http://178.154.192.57:8080

# Запустите тесты
go test -v
```

## Особенности

### Автоматическое управление тестовыми данными

- **Не требуется ручная настройка**: Тесты создают своих пользователей, хакатоны и данные
- **Уникальные данные для каждого теста**: Каждый тест генерирует уникальные email и username используя UUID
- **Нет конфликтов**: Тесты полностью изолированы и могут выполняться в любом порядке
- **Чистое состояние**: Каждый тест начинается с нуля без зависимости от предыдущих тестов

### Вспомогательные функции

Файл `setup_test.go` предоставляет:

- `NewTestContext(t)`: Создаёт контекст теста с HTTP клиентом
- `RegisterUser()`: Регистрирует нового пользователя и возвращает credentials с токенами
- `Login(creds)`: Аутентифицирует пользователя и обновляет access token
- `RegisterAndLogin()`: Комбинированная регистрация + вход
- `DoRequest()`: Выполняет HTTP запрос с опциональным телом и заголовками
- `DoAuthenticatedRequest()`: Выполняет аутентифицированный запрос с Bearer токеном
- `ParseJSON()`: Парсит JSON ответ в map
- `GenerateUniqueEmail()`: Генерирует уникальный email для тестирования

### Паттерны тестов

```go
func TestExample(t *testing.T) {
    tc := NewTestContext(t)
    
    // Автоматическая регистрация пользователя
    user := tc.RegisterUser()
    
    // Выполнение аутентифицированного запроса
    resp, body := tc.DoAuthenticatedRequest("GET", "/v1/users/me", user.AccessToken, nil)
    require.Equal(t, http.StatusOK, resp.StatusCode)
    
    // Парсинг и проверка ответа
    data := tc.ParseJSON(body)
    assert.Equal(t, user.UserID, data["user"].(map[string]interface{})["userId"])
}
```

## Интеграция с CI/CD

### Пример для GitHub Actions

```yaml
name: Integration Tests

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Set up Go
        uses: actions/setup-go@v4
        with:
          go-version: '1.24'
      
      - name: Start services
        run: docker compose -f deployments/docker-compose.yml up -d
      
      - name: Wait for API
        run: |
          timeout 60 bash -c 'until curl -f http://localhost:8080/swagger/ > /dev/null 2>&1; do sleep 2; done'
      
      - name: Run integration tests
        working-directory: tests/integration
        run: go test -v -parallel 4
      
      - name: Stop services
        if: always()
        run: docker compose -f deployments/docker-compose.yml down
```

## Устранение неполадок

### Connection Refused

**Проблема**: Тесты падают с ошибкой "connection refused"

**Решение**:
```bash
# Проверьте, что API запущено
curl http://localhost:8080/swagger/

# Проверьте Docker контейнеры
docker compose -f deployments/docker-compose.yml ps

# Посмотрите логи
docker compose -f deployments/docker-compose.yml logs gateway
```

### 401 Unauthorized

**Проблема**: Тесты падают с ошибками 401

**Решение**: Токен может быть просрочен или невалиден. Тесты автоматически обрабатывают обновление токенов, но проверьте:
```bash
# Проверьте, что auth service работает
curl -X POST http://localhost:8080/v1/auth/register \
  -H "Content-Type: application/json" \
  -d '{"username":"test","email":"test@example.com","password":"Test123","first_name":"Test","last_name":"User","timezone":"UTC"}'
```

### Тесты зависают

**Проблема**: Тесты зависают или превышают таймаут

**Решение**:
```bash
# Увеличьте таймаут
go test -v -timeout 5m

# Проверьте здоровье API
curl http://localhost:8080/swagger/
```

### Проблемы с базой данных

**Проблема**: Тесты падают с ошибками базы данных

**Решение**:
```bash
# Сбросьте базу данных
docker compose -f deployments/docker-compose.yml down -v
docker compose -f deployments/docker-compose.yml up -d

# Подождите миграций
sleep 10
```

## Лучшие практики

1. **Всегда используйте уникальные данные**: Используйте `tc.RegisterUser()` и `tc.GenerateUniqueEmail()` чтобы избежать конфликтов
2. **Независимость тестов**: Каждый тест должен быть запускаемым изолированно
3. **Чёткие проверки**: Используйте `require` для критических проверок, `assert` для некритических
4. **Осмысленные имена**: Имена тестов должны чётко описывать что они тестируют
5. **Позитивные и негативные кейсы**: Тестируйте как успешные, так и провальные сценарии
6. **Проверка побочных эффектов**: После операций проверяйте, что состояние изменилось как ожидалось
